Requires Changes

1 specification requires changes

Implement a basic driving agent

Student is able to implement the desired interface to the agent that accepts specified inputs.

The driving agent produces a valid output (one of None, ‘forward’, ‘left’, ‘right’) in response to the inputs.

Since the environment assigns a penalty to each negative action, such as going forward on a red light or when other cars are coming, the current learner keeps getting penalized and is likely to never reach the destination.

Good job describing behavior of agent when action is set to random. You're absolutely right that the agent does not learn, at this point it simply moves about randomly without any regards to the traffic rules.

The driving agent runs in the simulator without errors. Rewards and penalties do not matter - it’s okay for the agent to make mistakes.

Identify and update state

Student has identified states that model the driving agent and environment, along with a sound justification.

Good job selecting the states that you have. You're right deadline wouldn't be a good state space (increased complexity) without adding much value and we would ideally like the agent to move towards the destination while obeying traffic rules regardless of the time.

While ignoring left can cause the agent to learn that taking a left turn on a red light is an permitted action, we will assume that ideally this will rarely happen in the current simulation.

Do re-consider excluding the 'left' state from the model, doing this in essence means that we're choosing to not take into account the left state, which would make the model far from optimal. While it's true that it will rarely happen, bear in mind this is a cab and we should wish to make it as safe as possible. Also I can tell you it is absolutely required to include 'Left' to meet specifications for this section.

If we multiply each of the following values by each other => 3 x 2 x 4 x 2 , we get 48. This will constitute all the possible combinations between the state and the actions for our agent to add to its QTable

The multiplication done here includes the number of actions as well, only the total number of states should be reported here.

The driving agent updates its state when running, based on current input. The exact state does not matter, and need not be correlated with inputs, but it should change during a run.

Agent updates it's state and the display reflects this too.

Implement Q-Learning

The driving agent updates a table/mapping of Q-values correctly, implementing the Q-Learning algorithm.

Great job on your implementation of Q-Learning.

Given the current set of Q-values for a state, it picks the best available action.

Model picks the best available action. Also great work including the tie breaker when more than one. Good job on your implementation of the epsilon greedy strategy, a more advanced method to try is the epsilon decay, where epsilon is decreased over time to encourage less exploration and more exploitation.

Student has reported the changes in behavior observed, and provided a reasonable explanation for them.

Excellent work here. A discussion was included here on the agent's behavior as well as provided an explanation for why this is, this was also compared to the random implementation and the performance of the model included.

Enhance the driving agent

The driving agent is able to consistently reach the destination within allotted time, with net reward remaining positive.

Agent consistently reaches destination

Specific improvements made by the student beyond the basic Q-Learning implementation have been reported, including at least one parameter that was tuned along with the values tested. The corresponding results for each value are also reported.

This is really good work! Multiple parameter combinations are tried and the results displayed as a table. I also like that you included further discussions on each of the experiments. Awesome job!

Also note that while the success rate metric does capture the idea of how well the model is performing in terms of reaching the destination, it doesn't quite capture how well it's performing with regards to the agent following traffic rules, a useful metric to add here would be something like the total penalties incurred.

A description is provided of what an ideal or optimal policy would be. The performance of the final driving agent is discussed and compared to how close it is to learning the stated optimal policy.

Great job! An optimal policy is described and a thorough discussion was included on the solution.
